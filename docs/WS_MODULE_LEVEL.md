# WS_MODULE_LEVEL: руководство по подключению модуля к веб-сокету

Этот документ объясняет, что именно модуль может и должен делать через
`lcmm-ws`.

## Главная идея

Модуль не должен напрямую владеть websocket-транспортом.

Он владеет только:

1. смыслом своих событий;
2. правилами видимости подписок;
3. переводом внутреннего события модуля в сообщение для браузера.

Все остальное остается за приложением.

## Что модуль может регистрировать

В `init!` модуль может зарегистрировать:

1. handler для своих topic-ов;
2. функцию `authorize-subscribe`;
3. функцию, которая превращает внутреннее событие в тему и сообщение.

При этом модуль не обязан знать:

1. как именно приложение разбирает входящее сообщение браузера;
2. как приложение приводит тему к своей внутренней форме;
3. каким вспомогательным слоем приложение обрабатывает подписку и отписку.

Модуль получает уже устойчивую форму темы и сессии.

## Что модуль не должен делать

1. поднимать свой websocket endpoint;
2. управлять raw channel;
3. менять handshake policy;
4. менять `Origin` policy;
5. подменять собой общую проверку входа в приложение;
6. забирать себе жизненный цикл транспортного слоя.

## Как выглядит handler подписки

Идея простая:

1. модуль описывает, какие темы он понимает;
2. модуль проверяет, допустима ли подписка для конкретной сессии;
3. библиотека и приложение уже делают саму транспортную работу.

Именно так и должно быть: модуль решает вопрос доступа, а не вопрос
транспортной механики.

Примерная форма:

```clojure
{:module :booking
 :topic-kind :user
 :match-topic? (fn [topic]
                 (and (vector? topic)
                      (= :user (first topic))))
 :authorize-subscribe
 (fn [{:keys [session topic]}]
   {:ok? (= (get-in session [:subject :user-id])
            (second topic))})}
```

## Как выглядит projector

Эта функция получает модульное событие и возвращает, что надо отправить в браузер.

Он не должен сам отправлять сообщение.

Примерная форма:

```clojure
{:module :booking
 :event-type :booking/created
 :project
 (fn [{:keys [envelope]}]
   [{:topic [:user (get-in envelope [:payload :user-id])]
     :message {:type "event"
               :event "booking/created"
               :payload {:bookingId (get-in envelope [:payload :booking-id])}}}])}
```

## Практическое правило

Если модулю не хватает данных для правильной работы через веб-сокет, значит
нужно расширять модульный контракт, а не лезть из корневой сборки приложения в
чужие внутренности.

## Что это дает

Такая форма позволяет:

1. мыслить модуль как отдельный проект;
2. держать предметные знания рядом с модулем;
3. подключать и удалять модуль без переписывания транспортной склейки;
4. сохранить управление websocket-контуром за приложением.
