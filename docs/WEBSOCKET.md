# WEBSOCKET: app-level доставка событий в браузер

Версия: `1.0-draft`

Этот документ описывает минимальный практический подход к веб-сокету в LCMM.

Главная идея:

веб-сокет в LCMM считается транспортом уровня приложения, а не новым способом
межмодульной связности.

Если нужен порядок app-level сборки, сначала смотрите
[APP_COMPOSITION](./APP_COMPOSITION.md).
Если нужен общий HTTP-контур и browser-first API, смотрите
[HTTP](./HTTP.md) и [HTTP_FRONTEND_NOTES](./HTTP_FRONTEND_NOTES.md).

## 1. Когда веб-сокет вообще нужен

Для малого и среднего проекта веб-сокет нужен не “на всякий случай”, а когда
действительно требуется push из приложения в браузер.

Практический базовый сценарий:

1. начальное состояние грузится по HTTP;
2. веб-сокет доставляет новые события;
3. после переподключения клиент при необходимости перечитывает состояние по HTTP.

Это означает:

1. веб-сокет не заменяет обычный HTTP API;
2. экран не должен зависеть только от постоянного сокет-соединения;
3. потеря соединения не должна делать поведение приложения непредсказуемым.

## 2. Что именно принадлежит уровню приложения

Уровень приложения владеет:

1. endpoint-ом веб-сокета;
2. хранением соединений и подписок;
3. проверкой доступа на уровне транспортного канала;
4. доставкой сообщения нужным подписчикам;
5. браузерной формой сообщения;
6. ручной demo-страницей, если она нужна примеру.

Уровень модуля владеет:

1. предметными событиями;
2. предметными инвариантами;
3. правилами изменения состояния;
4. каноническим смыслом данных;
5. своими контрактами чтения;
6. своим контрактом экспорта в веб-сокет, если приложение использует `ws-registry`.

Короткое правило:

приложение знает, **кому** и **как** доставить сообщение, но не должно решать,
**что именно означает событие** в предметной области.

## 3. Чего приложение делать не должно

Приложение не должно:

1. читать внутренние таблицы модуля ради веб-сокета;
2. вызывать внутренние функции модуля вне явного контракта;
3. само вычислять новый предметный смысл из внутренних полей модуля;
4. подменять модульные правила своими правилами “для браузера”.

Если для браузерной доставки не хватает данных или смысла, правильный путь такой:

1. сначала расширить модульный контракт;
2. затем использовать этот контракт на уровне приложения.

Практическое уточнение:

если используется `lcmm-ws`, модуль может в `init!` зарегистрировать:

1. обработчик своих подписок;
2. функцию, которая превращает внутреннее событие в сообщение для браузера.

Но от этого веб-сокет не становится модульным транспортом. Открытие соединения,
проверка `Origin`, проверка входа и жизненный цикл соединения все равно остаются
на стороне приложения.

## 4. Минимальный scope первой фазы

Для первого рабочего варианта LCMM рекомендует очень узкий scope:

1. `subscribe`
2. `unsubscribe`
3. `ping`
4. server push от приложения к браузеру

Первая фаза сознательно не включает:

1. произвольный command API по веб-сокету;
2. durable delivery;
3. offline queue;
4. обязательную отдельную библиотеку;
5. распределенное хранение соединений.

Это уменьшает сложность и поверхность атаки.

## 5. Базовая модель

Опорная схема:

```text
browser
-> websocket endpoint уровня приложения
-> транспортная проверка доступа
-> подписка на тему, принятую в приложении
-> модуль публикует событие в event-bus
-> общий связующий слой приложения доставляет сообщение подписчикам
```

Пример:

```clojure
;; условный связующий слой приложения, не публичный API библиотеки
;; модуль публикует событие
(bus/publish bus
             :booking/created
             {:booking-id booking-id
              :slot-id slot-id
              :user-id user-id}
             {:module :booking})

;; приложение доставляет его браузеру
(bus/subscribe bus
               :booking/created
               (fn [_ envelope]
                 (ws/send-to-topic! ws-hub
                                    {:topic [:user (get-in envelope [:payload :user-id])]
                                     :event "booking/created"
                                     :payload (:payload envelope)})))
```

## 6. HTTP и веб-сокет вместе

Рекомендуемая клиентская модель:

1. initial load по HTTP;
2. live update по веб-сокету;
3. reconnect;
4. переподписка;
5. повторное чтение состояния по HTTP.

Пример практического потока:

```text
GET /bookings?user-id=u-alice
-> connect /ws
-> subscribe ["user","u-alice"]
-> receive booking/created
-> disconnect
-> reconnect
-> GET /bookings?user-id=u-alice
```

## 7. Browser-facing контракт

Внутреннее модульное событие и браузерное сообщение не обязаны совпадать один в
один.

Приложение может:

1. убрать лишние поля;
2. перевести имена в более удобную внешнюю форму;
3. добавить технические поля вроде `correlationId`.

Приложение не должно:

1. менять предметный смысл сообщения;
2. придумывать новый предметный статус, которого модуль сам не объявлял.

Минимальные сообщения первой фазы:

```json
{"type":"subscribed","topic":["user","u-alice"]}
{"type":"unsubscribed","topic":["user","u-alice"]}
{"type":"pong"}
{"type":"error","code":"subscription_rejected","message":"Subscription rejected"}
{"type":"event","event":"booking/created","topic":["user","u-alice"],"payload":{"bookingId":"...","userId":"u-alice","slotId":"slot-09-00"},"correlationId":"..."}
```

## 8. Безопасность

Минимальный baseline первой фазы:

1. auth на handshake уровне приложения;
2. проверка `Origin` по явной политике;
3. запрет секретов в URL/query string;
4. authorization на каждый `subscribe`;
5. anti-enumeration для ошибок подписки;
6. reconnect проходит ту же проверку заново;
7. raw payload не логируется по умолчанию.

Пример безопасного отказа:

```json
{"type":"error","code":"subscription_rejected","message":"Subscription rejected"}
```

Практическое разделение ответственности:

1. `lcmm-guard` отвечает за IP-based rate limit, detector, ban и trusted proxy policy;
2. app-level websocket слой отвечает за `Origin`, размер сообщения, лимит подписок,
   лимит соединений, idle timeout и политику закрытия соединения.

Подробнее см. также [SECURITY](./SECURITY.md), [SECURE_APP](./SECURE_APP.md) и
[GUARD](./GUARD.md).

## 9. Operational baseline

Первая фаза должна быть предсказуемой и скромной.

Практические правила:

1. доставка считается best-effort;
2. медленный клиент может быть отключен;
3. при disconnect подписки теряются;
4. после reconnect клиент подписывается заново;
5. если событие произошло во время disconnect, доставка не обещается;
6. после reconnect клиент по умолчанию перечитывает нужные данные по HTTP.

## 10. Наблюдаемость

Минимум, который стоит видеть:

1. число активных соединений;
2. число подписок;
3. число входящих и исходящих сообщений;
4. rejected subscriptions;
5. malformed messages;
6. закрытия по политике.

Если нужен отдельный слой метрик и логов, смотрите [OBSERVABILITY](./OBSERVABILITY.md).

## 11. Ручная проверка в примере

Для `examples/booking-full` веб-сокет проверяется не из одной адресной строки, а
через встроенную страницу:

1. приложение отдает `/ws-demo`;
2. страница открывается обычным браузером;
3. на ней можно открыть соединение, подписаться и увидеть push-сообщения.

Важно:

это не отдельное фронтенд-приложение, а маленькая диагностическая страница,
нужная именно для ручной проверки transport flow.

## 12. Упрощение в `booking-full`

Текущая реализация `booking-full` использует упрощенную идентификацию demo-пользователя
через `user-id` в query string веб-сокет endpoint-а.

Это допустимо только как демонстрационное упрощение примера.

Для реального приложения рекомендуемый путь другой:

1. app-level auth middleware;
2. повторная проверка auth/authz на handshake;
3. отсутствие секретов в URL;
4. единая browser security policy вместе с остальным HTTP-контуром.

## 13. Когда думать об отдельной библиотеке

Отдельная библиотека нужна не по умолчанию, а только если:

1. один и тот же app-level код повторился минимум в двух местах;
2. уже видно стабильную общую модель;
3. вынос действительно уменьшает сопровождение, а не увеличивает его.

До этого этапа нормальным считается держать websocket wiring внутри приложения.

Практическая оговорка:

если в проекте используется `lcmm-ws`, разумно вынести повторяющуюся
транспортную механику и общее хранилище сессий и подписок в библиотеку, а
модульный экспорт оформлять
через `ws-registry`.

Это не меняет главный принцип документа:

1. websocket-транспорт остается на стороне приложения;
2. модуль не получает raw channel;
3. модуль лишь регистрирует свой контракт экспорта.
